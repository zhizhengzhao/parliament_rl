# RL pipeline

Three scripts, one per step:

| script | in | out | purpose |
|---|---|---|---|
| `rl/extract.py` | `parliament.db` | `train.jsonl` | rebuild context, compute reward/advantage |
| `rl/train.py` | `train.jsonl` + base model | `ckpt/step_K/` | FSDP2 offline RL (GRPO/RWR + ratio + KL) |
| `rl/export.py` | `ckpt/step_K/` | `merged/` | gather sharded checkpoint → vLLM-loadable dir |

`scripts/iterate.py` chains them into a resumable multi-shard loop.

## Extraction — what a training sample looks like

Each actor post becomes one training example of the form

```json
{
  "messages": [
    {"role": "user", "content": "You are Alice, ...\n\n## Problem\n...\n\n## Discussion\n[P_1] by Rafael (score: +5)\n...\n\n## Your Next Contribution"},
    {"role": "assistant", "content": "<the post content the actor actually produced>"}
  ],
  "reward": 6.0,
  "advantage": 1.05,
  "session_id": "abc12345",
  "post_id": 42,
  "author": "Alice"
}
```

### Why natural language (not agent transcripts)

The training sample is a plain user/assistant chat, not a replay of
the agent's tool-call transcript. We do not include system prompts,
persona text, `<tool_call>` blocks, or `python_exec` outputs. The
goal is to train **scientific reasoning**, not the agent format —
what the model learns must transfer to any context, not just
Parliament's.

### Reward = judge votes only

`reward_i = sum of judge votes cast on post i`. Actor votes are still
visible in the context (as `(score: +N)` suffixes, when enabled) —
the model reads them as signal — but they do not enter the reward
arithmetic. The reasoning:

- Judge votes are grounded in the reference solution. Actor votes are
  peer guesses from agents who themselves are trying to solve the
  problem. Mixing them would introduce correlated noise in the reward.
- Keeping actors in the CONTEXT preserves the social-dynamics signal
  (what peers thought) without biasing the TRAINING signal.

### Advantage normalization

```
A_i = (r_i − baseline) / scale
```

Both `baseline` and `scale` are configured in
`context_configs/RL_context/config.json`:

- `advantage_baseline`: `0.0` (default, preserves absolute reward
  sign), `"mean_session"` (classic GRPO centering), `"mean_global"`
  (REINFORCE++-style), or any number.
- `advantage_scale`: `"session_std"` (default; per-session std with a
  1.0 floor so a homogeneous session can't blow up), `"global_std"`
  (REINFORCE++-style), `"none"`, or any number.

The default (`baseline=0`, `scale=session_std`) keeps the *absolute*
judge signal intact — a post judged +6 always has positive advantage,
regardless of how bright its session peers were — while normalizing
the magnitude against local variation. Posts in harder sessions
(where everyone got low scores) don't dominate gradients just because
their raw rewards are lower than an easy session's.

### Identity anonymization & score visibility

Two knobs designed for ablations:

- `anonymize_identity: true/false`. When true (default), the same
  session-seeded name map that the harness used at generation time is
  applied when rendering training headers, so the model reads a
  discussion between "Alice", "Rafael", "Ivan", not
  "Scientist_1/2/3". The map is drawn from a 406-entry flat pool in
  `Parliament_context/config.json` (single source of truth,
  bit-identical between harness and extract).

- `score_visibility: "auto" | "always" | "never"`. `auto` keeps the
  `(score: ±N)` suffixes only for sessions whose actor posts
  explicitly reference parliament scoring — roughly 35% of sessions
  in practice. The remaining 65% of sessions, which never mention
  scores in their content, have score-free headers. This eliminates
  the train/inference distribution shift for the bulk of samples
  without erasing useful within-context signal where it is natural.

### Extraction cost

Single pass over `parliament.db`:

- `O((P + C + V) · log V)` per session for vote folding (sort votes by
  time once, streaming sweep through the timeline).
- Global normalization statistics are computed in a pre-pass over all
  sessions.

A 1026-session / ~10 k sample DB extracts in ~45 seconds on CPU.

## Training — GRPO with ratio + clip

### Loss

```
L = −min( ρ · A,  clip(ρ, 1-ε, 1+ε) · A )   +   β · KL(π || π_ref)
```

where

- `ρ = π_θ(a|s) / π_old(a|s)` is the importance ratio (sequence-level,
  length-normalized).
- `A` is the precomputed advantage from `extract.py`.
- `ε = 0.2` (PPO-standard clip).
- KL uses the Schulman k3 estimator in fp32 with a log-ratio clamp at
  ±10 to prevent BF16 `exp()` overflow.
- `β = 0.001` by default (gentle anchor to the reference).

With `--use-ratio` off, the loss degenerates to `−A · log π(a|s) +
β · KL` — the RWR objective. This is equivalent to ratio + clip
at step 0 where π = π_old and ρ = 1. We default ratio ON because we
train ≥ 2 epochs per iteration, and from epoch 2 onwards π has
drifted enough that the clip matters.

### Implementation

One file: `rl/train.py` (~460 lines). Single GPU for development,
FSDP2 via `accelerate` for production. Uses:

- Activation checkpointing, BF16 compute with fp32 master weights,
  sharded state-dict checkpoints.
- `Qwen3_5DecoderLayer` as the FSDP wrap class.
- Hard-exit via `os._exit(0)` after final save to bypass a known NCCL
  teardown deadlock in the Python interpreter shutdown path.
- Response masking that locates the last `<|im_start|>` → subsequent
  `</think>\n\n` to mark the response span, robust to user content
  that happens to contain `</think>` literals.
- Advantage-clip, grad-clip, checkpoint rotation (`--keep-last N`) and
  clean resume from any sharded checkpoint.

### Reference-model placement

The reference model is loaded full-replicated on every GPU, not
FSDP-sharded. At 9B BF16 this is 18 GB / GPU — comfortable on
80 GB cards, and avoids FSDP2's "prepare(model, optimizer) as a pair"
constraint that doesn't accommodate a frozen no-gradient model.

## Iterative schedule — `scripts/iterate.py`

A single collection of N rollouts is fresh only against the π that
produced it. Once training moves π, the remaining rollouts are
progressively off-policy. Ratio + clip bound this drift within a
bounded range, but the dataset's effective learning signal decays.

The remedy is **sharding the corpus and rotating through
iterations**:

```
iter k:  sample shard_k with π_{k−1}  →  extract  →  train  →  π_k
iter k+1: sample shard_{k+1} with π_k →  extract  →  train  →  π_{k+1}
```

Natural sharding for the Sciencepedia dataset is the 4 × 1026-question
split in `datasets/sciencepedia_train_part{1..4}.json`:

```
iter 1: part1 (π_0 = Qwen3.5-9B)  →  π_1
iter 2: part2 (π_1)               →  π_2
iter 3: part3 (π_2)               →  π_3
iter 4: part4 (π_3)               →  π_4
```

On 8 × A100-80GB each iteration is ≈ 13.5 h
(9.3 h sampling + 4 h training + small overhead). Full arc
≈ 54 h.

### Why this shard size

- Smaller shards (e.g. 16 × 256 questions): per-iteration step count
  drops to ~40, signal-to-noise becomes poor; also vLLM startup
  overhead proportionally grows.
- Larger shards / no sharding: the second epoch of a huge batch
  already sees a very drifted π; ratio clip saturates; effective
  learning rate shrinks; iteration count is lower but wall clock per
  useful gradient step is worse.
- 1026-question shards ~ 140 optimizer steps per iteration: enough
  statistical power per iteration, clean iteration boundaries,
  natural dataset split.

### Resume & crash recovery

`scripts/iterate.py` persists state after every iteration to
`data/<name>_<ts>/state.json` (`completed`, `current_model`,
`history`). Re-invoking the same command picks up from the last
completed iteration. Sharded checkpoints are pruned after each
iteration's merged export (saves ~51 GB per iteration); the merged
HF folder is all we need to roll forward.

## Hyperparameters (defaults; all overridable via CLI or `--train-extra`)

| knob | default | notes |
|---|---|---|
| `learning_rate` | 1e-6 | DeepSeek-R1's Qwen-9B default |
| effective batch | 128 | 1 × 16 grad-accum × 8 GPUs |
| `num_epochs` | 2 | per iteration |
| `warmup_steps` | 10 | cosine decay after |
| `grad_clip` | 1.0 | LLM-RL standard |
| `beta_kl` | 0.001 | gentle anchor to previous iteration |
| `use_ratio` | true | PPO ratio + clip required for >1 epoch |
| `advantage_clip` | 0 (off) | advantages already bounded |
| `save_every` | 50 | steps |
| `keep_last` | 3 | checkpoint rotation |
| `max_seq_len` | 8192 | drops ~10% of samples > 8 k tokens |

All of these are easy to override per-experiment — see
`--train-extra "--beta-kl 0.01 --num-epochs 3"` in
`scripts/iterate.py`.
