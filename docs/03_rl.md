# RL pipeline

Three scripts, one per step:

| script | in | out | purpose |
|---|---|---|---|
| `rl/extract.py` | `parliament.db` | `train.jsonl` | rebuild context, compute reward/advantage, debias position |
| `rl/train.py` | `train.jsonl` + base model | `ckpt/step_K/` | DDP + LoRA + RWR + KL anchor |
| `rl/export.py` | `ckpt/step_K/` | `merged/` | merge LoRA into base + patch visual/mtp weights |

`scripts/iterate.py` chains them into a resumable, **per-step idempotent**
multi-shard loop.

## Extraction — what a training sample looks like

Each *actor* produces **one training sample per session** containing
the actor's full multi-turn trajectory:

```json
{
  "messages": [
    {"role": "user",      "content": "...problem + initial discussion..."},
    {"role": "assistant", "content": "...the actor's 1st post..."},
    {"role": "user",      "content": "...updates since the 1st post..."},
    {"role": "assistant", "content": "...the actor's 2nd post..."},
    ...
  ],
  "turn_rewards":    [r1, r2, ...],     // one per assistant turn
  "turn_advantages": [a1, a2, ...],     // one per assistant turn
  "turn_post_ids":   [pid1, pid2, ...],
  "session_id":      "abc12345",
  "actor":           "Alice"
}
```

The trainer's `make_response_mask` returns both `mask` and `segments`
(turn indices), and the per-turn advantage is broadcast to every
response token of that turn via the segment mask. Samples with fewer
than 2 turns are dropped (the multi-turn structure is what makes the
KL anchor stable across iters).

### Why natural language (not agent transcripts)

The training sample is a plain user/assistant chat, not a replay of
the agent's tool-call transcript. We do not include system prompts,
persona text, `<tool_call>` blocks, or `python_exec` outputs. The
goal is to train **scientific reasoning**, not the agent format —
what the model learns must transfer to any context, not just
Parliament's.

### Reward = judge votes only

`reward_i = sum of judge votes cast on post i`. Actor votes are still
visible in the context (as `(score: +N)` suffixes, when enabled) —
the model reads them as signal — but they do not enter the reward
arithmetic. The reasoning:

- Judge votes are grounded in the reference solution. Actor votes are
  peer guesses from agents who themselves are trying to solve the
  problem. Mixing them would introduce correlated noise in the reward.
- Keeping actors in the CONTEXT preserves the social-dynamics signal
  (what peers thought) without biasing the TRAINING signal.

### Advantage normalization

```
A_i = (r_i − baseline) / scale
```

Both `baseline` and `scale` are configured in
`context_configs/RL_context/config.json`:

- `advantage_baseline`: `"mean_session"` (default; classic GRPO
  centering), `0.0` (preserves absolute reward sign — a +6 post stays
  positive regardless of session mean), `"mean_global"` (REINFORCE++-
  style), or any number.
- `advantage_scale`: `"session_std"` (default; per-session std with a
  1.0 floor so a homogeneous session can't blow up), `"global_std"`
  (REINFORCE++-style), `"none"`, or any number.

The default (`baseline="mean_session"`, `scale="session_std"`) is the
standard GRPO recipe — within each session, posts are scored relative
to that session's mean and normalized by its std. Posts in harder
sessions (where everyone got low scores) don't dominate gradients
just because their raw rewards happen to be lower than an easier
session's. The `mean_session` baseline can change to `0.0` if you
want the absolute judge sign preserved (useful when you trust the
judge's calibration across sessions).

### Position debiasing

A linear "later posts score higher" trend exists across the corpus —
late posts build on already-validated chains and naturally collect
positive judge votes more often. Without correction the model would
learn "stay silent until late = good".

`extract.py:debias_position` estimates the per-corpus slope from the
gap between the first-half and second-half mean rewards (midpoints at
t≈0.25 and t≈0.75) and removes it:

```
r' = r - slope * (t - 0.5)
```

where `t` ∈ [0, 1] is the post's normalized position in its session.
Skipped when `|slope| < 0.01` so a flat corpus pays no penalty.
Position-debiased rewards then feed advantage normalization.

### Identity anonymization & score visibility

Two knobs designed for ablations:

- `anonymize_identity: true/false`. When true (default), the same
  session-seeded name map that the harness used at generation time is
  applied when rendering training headers, so the model reads a
  discussion between "Alice", "Rafael", "Ivan", not
  "Scientist_1/2/3". The map is drawn from a 406-entry flat pool in
  `Parliament_context/config.json` (single source of truth,
  bit-identical between harness and extract).

- `score_visibility: "auto" | "always" | "never"`. `auto` keeps the
  `(score: ±N)` suffixes only for sessions whose actor posts
  explicitly reference parliament scoring — roughly 35% of sessions
  in practice. The remaining 65% of sessions, which never mention
  scores in their content, have score-free headers. This eliminates
  the train/inference distribution shift for the bulk of samples
  without erasing useful within-context signal where it is natural.

### Extraction cost

Single pass over `parliament.db`:

- `O((P + C + V) · log V)` per session for vote folding (sort votes by
  time once, streaming sweep through the timeline).
- One pre-pass collects per-session rewards for global-stats /
  position-slope estimation; second pass renders samples.

A 1026-session / ~10 k sample DB extracts in ~45 seconds on CPU.

## Training — token-level RWR + KL anchor

### Loss

```
L = −mean_token( clip(A, ±C) · logπ_θ(y_t|·) )       ← policy gradient
   + β · mean_token( exp(log r) − log r − 1 )         ← KL anchor (k3)
        where  log r = logπ_θ − logπ_base
              base   = LoRA disabled (cheap π_ref)
```

`mean_token = (x * mask).sum() / mask.sum()` over the whole batch
(DAPO-style) so every response token contributes equally — no length
bias, no per-sequence renormalization that would over-weight short
rollouts.

### Why no PPO ratio

`rl/extract.py` re-renders every prompt from `parliament.db`
(anonymized names, custom headers, `[you]` markers, score-visibility
gating). The training prompt-token sequence therefore differs from
what the rollout actually saw, so any importance ratio against a
logged "old log-prob" would be against the wrong distribution. Treat
the data as off-policy and use the off-policy-correct objective:
weighted regression. Multi-iter safety comes from the KL anchor
to base plus ReST-style data refresh across iters / total_epochs in
`scripts/iterate.py`.

### Reference-model placement (LoRA trick)

The reference model is the **same model with the LoRA adapter
disabled** (`PeftModel.disable_adapter()`). No second 18 GB copy is
needed — the inactive base is reused for both policy and reference
forwards. This works only on the LoRA path; the legacy full-FT path
(`--no-use-lora`) loads a separate frozen `ref_model`.

### Multi-turn loss assignment

`make_response_mask` walks the input ids, finds every
`<|im_start|>...assistant\n<think>...</think>\n\n` block, and emits
two tensors:

- `mask`: 1 on every assistant *response* token (post-`</think>`,
  pre-`<|im_end|>`), 0 elsewhere.
- `segments`: turn index (1, 2, 3, …) for each response token, 0
  elsewhere.

Per-turn advantages are then expanded into a `[B, T]` per-token
advantage tensor. The loss is one big `mean_token` over the whole
mask — which is mathematically equivalent to weighted-mean across
turns, with weights proportional to turn length.

### Implementation notes

- Single file: `rl/train.py` (~700 lines). DDP via `accelerate` is
  default; FSDP yaml exists as fallback for `--no-use-lora` only.
- BF16 end-to-end. **No `mixed_precision="bf16"`** because that
  inserts an autocast wrapper that casts logits BF16→FP32 at the
  boundary — for vocab=152k and seq=8k that's ~6 GB extra per
  forward, and we run two forwards (policy + ref).
- `per_token_log_p` chunks the time dim by 1024 to keep peak memory
  at `O(B · chunk · V)`. Same trick used by Verl / Megatron-LM for
  the LM-head softmax.
- KL k3 estimator: `log_r` clamped to ±10 before `exp()` to avoid
  BF16 overflow on the rare long-sequence outlier.
- `LengthGroupedSampler` sorts samples by length and packs
  similar-length items into batches — minimal padding waste, and the
  batch *order* is reshuffled each ppo epoch.
- Activation checkpointing on; LoRA path also calls
  `enable_input_require_grads()` so frozen-embedding outputs
  propagate gradients through the adapter.
- `os._exit(0)` after final save to bypass a known NCCL teardown
  deadlock in the Python interpreter shutdown path.

## Iterative schedule — `scripts/iterate.py`

Three nested loops (verl-aligned, see [`00_naming.md`](00_naming.md)):

```
total_epoch = 1..E             # outer: full pass through the shard list
  iter      = 1..N_shards × E  # middle: one shard per iter
    ppo_epoch = 1..K           # inner: passes through this iter's train.jsonl
```

A single collection of N rollouts is fresh only against the π that
produced it. Once training moves π, the remaining rollouts are
progressively off-policy. RWR's weighted-regression objective is
off-policy-correct in expectation but variance grows as π drifts.
The remedy is **sharding the corpus and rotating through iters**:

```
iter k:  sample shard_k with π_{k−1}  →  extract  →  train  →  π_k
iter k+1: sample shard_{k+1} with π_k →  extract  →  train  →  π_{k+1}
```

Natural sharding for the Sciencepedia dataset is the 4 × 1026-question
split in `datasets/sciencepedia_train_part{1..4}.json`. With
`--total-epochs 2` the schedule becomes 8 iters:

```
ep1.iter1: part1 (π_0 = Qwen3.5-9B)  →  π_1
ep1.iter2: part2 (π_1)               →  π_2
ep1.iter3: part3 (π_2)               →  π_3
ep1.iter4: part4 (π_3)               →  π_4
ep2.iter5: part1 (π_4)               →  π_5     ← second pass already starts strong
ep2.iter6: part2 (π_5)               →  π_6
ep2.iter7: part3 (π_6)               →  π_7
ep2.iter8: part4 (π_7)               →  π_8
```

The KL anchor stays pinned to the original Qwen3.5-9B for every iter
(`--ref-model` is fixed); only the rollout policy changes.

On 8 × A100-80GB one iter is ≈ 1.5 h on mid200 (200 questions/shard);
full 1026-question shards take ≈ 14 h (9 h sampling + 4 h training).

### Why this shard size

- Smaller shards (e.g. 16 × 256 questions): per-iter step count
  drops to ~40, signal-to-noise becomes poor; also vLLM startup
  overhead proportionally grows.
- Larger shards / no sharding: the second ppo epoch of a huge batch
  already sees a very drifted π; KL anchor saturates; effective
  learning rate shrinks; iter count is lower but wall clock per
  useful gradient step is worse.
- 1026-question shards ~ 140 optimizer steps per iter: enough
  statistical power per iter, clean iter boundaries, natural dataset
  split.

### Resume & crash recovery — per-step idempotent

`scripts/iterate.py` persists state after every iter to
`data/<name>_<ts>/state.json` (`completed`, `current_model`,
`history` — each history entry tagged with `total_epoch`). Re-invoking
the same command picks up from the next incomplete iter.

**Within an interrupted iter**, every sub-step is also idempotent —
re-run the same command and:

| step | skip condition |
|---|---|
| sample | `parliament.db` + `experiment.json` already in run dir |
| extract | `train.jsonl` exists and > 100 bytes |
| train | auto `--resume step_K` for the highest existing checkpoint |
| export | `merged/` has `config.json`, ≥1 safetensors shard, and `tokenizer_config.json` |

So a crash mid-training only re-does work from the last surviving
checkpoint, and a crash mid-sample re-uses the partial DB on next run
only if it was actually finished (we require `experiment.json` as the
"sampling completed" sentinel).

Sharded checkpoints are pruned after each iter's merged export
(saves ~51 GB per iter); the merged HF folder is all we need to
roll forward.

## Hyperparameters (defaults; all overridable via CLI or `--train-extra`)

| knob | default | notes |
|---|---|---|
| `learning_rate` | 1e-4 | LoRA rate; lower (~1e-6) for full FT |
| effective batch | 128 | 1 × 16 grad-accum × 8 GPUs |
| `ppo_epochs` | 2 | 1–2 is plenty for RWR (verl: `actor.ppo_epochs`); CLI `--ppo-epochs`, alias `--num-epochs` |
| `warmup_steps` | 10 | cosine decay after |
| `weight_decay` | 0.0 | RL fine-tuning doesn't benefit |
| `grad_clip` | 1.0 | LLM-RL standard |
| `beta_kl` | 0.02 | gentle anchor to original base; raise to 0.05 if KL > 5, lower to 0.005 if PG signal suffocated |
| `advantage_clip` | 2.0 | clamp \|A\| ≤ 2 (covers ~95% of N(0,1) tail) |
| `lora_r` / `lora_alpha` | 32 / 64 | sweet spot at 9B |
| `lora_target_modules` | `q_proj,v_proj,o_proj` | attention-only — keeps base MLP/mamba intact |
| `save_every` | 50 | optimizer steps |
| `keep_last` | 3 | checkpoint rotation |
| `max_seq_len` | 8192 | clean truncation at user-turn boundary; safety against OOM |

All of these are easy to override per-experiment — see
`--train-extra "--beta-kl 0.01 --ppo-epochs 3"` in
`scripts/iterate.py`.
